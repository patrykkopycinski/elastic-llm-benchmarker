Benchmarking methodology

How models are benchmarked

Every model goes through a rigorous, multi-stage evaluation pipeline that tests real-world production readiness — not just synthetic scores. The pipeline runs autonomously on dedicated GPU hardware via SSH, ensuring consistent and reproducible results.

1
Model discovery & card parsing
The model's HuggingFace card is fetched and parsed to extract architecture, parameter count, context window, quantization options, supported tasks, and tool-calling capabilities. This metadata drives all downstream configuration decisions.
2
GPU deployment via vLLM
The model is deployed on a real GPU VM using vLLM inside a Docker container. Tensor parallelism, context length, and quantization are auto-configured based on the model card and hardware profile. The exact docker run command is recorded for reproducibility.
3
Stage 1 — Performance benchmarks
The model is stress-tested at multiple concurrency levels (1, 4, 16 concurrent requests). Key metrics are measured: Inter-Token Latency (ITL), Time To First Token (TTFT), throughput (tokens/sec), and P99 latency. Results are compared against size-tiered thresholds.
4
Quality gate
Models must pass performance thresholds before advancing: ITL p50 must be under size-based limits (20ms for ≤14B, 40ms for ≤40B, 60ms for ≤80B), throughput above 10 tok/s, TTFT under 5s, and context window ≥128K tokens. Models that fail are rejected with specific reasons.
5
Stage 2 — Kibana eval suites
Models that pass the quality gate are tested against Kibana's own evaluation suites. The Kibana repo is cloned, bootstrapped, and the model is evaluated on real product tasks — the same tests Elastic uses internally to validate LLM integrations.
6
Stage 3 — LLM reasoning analysis
An expert LLM (e.g., Claude) analyzes the full benchmark run: performance traces, eval results, deployment config, and failure modes. It produces actionable optimization suggestions (vLLM flags, quantization, hardware) that feed back into the next benchmark run.
7
Verdict & recommendation report
All evidence is synthesized into a final verdict: Support (production-ready), Investigate (potential with caveats), or Reject (does not meet requirements). Confidence is computed from the depth of evidence gathered across all stages.

What metrics are measured

  • ITL (Inter-Token Latency) — time between successive tokens; measures streaming responsiveness
  • TTFT (Time To First Token) — time from request to first token; measures perceived startup speed
  • Throughput (tok/s) — total tokens generated per second; measures raw generation capacity
  • P99 latency — 99th percentile end-to-end latency; measures worst-case user experience
  • Tool-calling success rate — correctness of structured tool calls in auto and required modes
  • Eval suite scores — pass rate on Kibana's product-level evaluation suites
  • GPU VRAM utilization — memory footprint on the deployed hardware

Why this is the right way to benchmark

Real hardware, not simulations
Models run on the same GPU VMs customers use. No synthetic estimates or paper benchmarks — actual Docker containers with measured GPU utilization, real network latency, and production-like memory pressure.
Multi-concurrency stress testing
Testing at concurrency 1/4/16 reveals how models degrade under load — a model fast at concurrency 1 may collapse at 16. Single-user benchmarks hide this critical production failure mode.
Product-level evaluation, not just metrics
Stage 2 uses Kibana's actual eval suites. This tests whether the model works for the product, not just whether it generates fast tokens. A model with great ITL but poor tool-calling is caught here.
Size-tiered thresholds
A 7B model is judged differently from a 70B model. ITL thresholds scale with parameter count because larger models are inherently slower but bring greater capability. Flat thresholds would unfairly reject capable large models.
Closed-loop optimization
Stage 3's reasoning feedback flows back into the next benchmark run. If the LLM suggests --gpu-memory-utilization 0.95 or --max-num-seqs 8, the next deployment applies those flags automatically. Models get a fair second chance.
Full reproducibility
Every benchmark records the exact Docker command, vLLM version, hardware config, tensor parallel size, and all flags. Any result can be reproduced by a customer with a single copied command.
