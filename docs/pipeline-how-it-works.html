Elastic LLM Benchmarker

An autonomous qualification pipeline for open-source LLMs targeting Elastic Agent Builder. It deploys models on a GPU VM via vLLM, measures performance, runs Kibana eval suites, reasons over traces, and produces a structured recommendation report — all orchestrated by a TypeScript daemon with Elasticsearch as the single source of truth.

1. What problem does it solve?

Product teams need empirical evidence before officially supporting a model in the Elastic Inference Service (EIS). Manually deploying, benchmarking, and evaluating each HuggingFace candidate takes days. The benchmarker automates that loop:

  • Discover promising models (optional HuggingFace polling)
  • Queue candidates that pass the Agent Builder baseline gate
  • Stage 1 — deploy + vllm bench serve performance gate
  • Stage 2 — Kibana @kbn/evals quality suites (local or Buildkite CI)
  • Stage 3 — LLM reasoning over metrics + OTel traces
  • Report — immutable support / investigate / reject verdict

2. System architecture

The daemon (benchmarker-queue start) runs a 30s (configurable) polling loop. All state — queue, results, traces, CI builds — lives in Elastic Serverless / local ES. The GPU VM is reached only through SSHClientPool; Buildkite agents reach the model through an ngrok tunnel.

Agent Builder baseline gate

Before a model is queued, evaluateAgentBuilderBaseline() checks requirements derived from elastic/security-team#15545:

  • ≥ 7B parameters (instruct variant)
  • ≥ 128K context window (enriched from HF config.json, including rope scaling)
  • Tool-calling support (single-tool via vLLM — not parallel/multi-tool)
  • Hardware fit for the configured VM profile (e.g. 2× A100-80GB)

4. Stage 1 — Performance gate

Stage 1 runs entirely on the GPU VM over SSH. If benchmarks fail thresholds, the pipeline stops here with a reject verdict — no expensive evals.

S1.1

Parse HuggingFace card

Architecture, param count, context window, tool-calling hints. Context is enriched from config.json (rope scaling → 131K for Qwen2.5).

S1.2

Deploy vLLM

Latest vllm/vllm-openai:latest, tensor-parallel across all GPUs, --max-model-len auto, Hermes tool parser when supported.

S1.3

Health check

Polls /v1/models until ready (up to 20 min). Classifies container logs on failure (OOM, disk, etc.).

S1.4

Benchmark suite

vllm bench serve at concurrency 1, 4, 16. Measures ITL p50, TTFT, throughput (tok/s), p99 latency.

S1.5

Threshold gate

Compares against benchmarkThresholds and stage2Thresholds: max ITL, min throughput, min context window.

S1.6

Persist + eligibility

Writes to benchmarker-results. Sets stage2Eligible — Stage 2 only runs if this is true.

Deployment stays warm. After Stage 1, the vLLM container remains running. The scheduler owns teardown — Stage 2 / CI evals reuse the same endpoint.

5. Stage 2 — Quality evals (two paths)

Stage 2 measures Agent Builder readiness via Kibana eval suites. The daemon supports two modes:

ModeWhenWhat runs
Buildkite CI (production) --ci-evals + Buildkite token Smoke test → on-demand Buildkite pipeline → @kbn/evals in Elastic CI
Local clone (dev) --stage2 without CI evals Clone Kibana → bootstrap → EvalSuiteRunner against local endpoint
When CI evals are enabled, the local Stage 2 worker is disabled — Buildkite results are mapped to the same Stage2Result shape via mapBuildkiteResultToStage2().

CI.1

SSH port forward

VllmPublicEndpointResolver always forwards VM port 8000 to laptop localhost — smoke tests never hit the firewalled public VM IP.

CI.2

ngrok tunnel

When TUNNEL_ENABLED=true, exposes localhost to a public HTTPS URL. Buildkite agents in Elastic CI need this to call your vLLM connector.

CI.3

Three-tier smoke test

Health → simple inference → tool-call validation. Fails fast before triggering a 30–60 min Buildkite build.

CI.4

Connector payload

Builds OpenAI-compatible connector JSON pointing at the ngrok URL. Passed as Buildkite env vars to the on-demand pipeline.

CI.5

On-demand Buildkite build

Pipeline: upload → pre-build → Kibana dist → LLM evals. Suites from buildkite.defaultEvalSuites (e.g. security-alert-triage).

CI.6

Detached poll

With detachPoll: true, poll runs in background while tunnel + model stay up. Scheduler waits before dequeuing the next model.

7. Stage 3 — LLM reasoning

After Stage 2 completes (sync or detached poll), Stage 3 uses Elastic Inference Service (e.g. eis/anthropic-claude-4.6-opus) to analyze the full run. 

flowchart LR
  IN["Inputs"] --> PB["ReasoningPromptBuilder"]
  PB --> LLM["EIS LLM client"]
  LLM --> OUT["JSON suggestions"]
  IN --- M1["Stage 1 metrics"]
  IN --- M2["Stage 2 suite scores"]
  IN --- M3["OTel trace summary\n(ES or local JSONL)"]
  OUT --> ES["benchmarker-reasoning"]
  OUT --> NEXT["Fed back to\nnext Stage 1 deploy"]
  • Queries traces via TraceQueryBuilder (ES|QL against OTel indices — PascalCase fields)
  • Builds a structured prompt with vLLM config, benchmark metrics, eval scores
  • Returns categorized suggestions: config, quantization, hardware, other
  • Stage 1 reads prior suggestions on re-runs and may apply vLLM deployment overrides
Stage 3 failures are non-fatal — the pipeline still completes and produces a report.

8. Recommendation report

buildRecommendationReport() synthesizes all stages into an immutable document stored in recommendation-reports.

support investigate reject
VerdictCondition
supportStage 1 passed AND all configured eval suites pass thresholds
investigateStage 1 passed but evals failed or near-threshold / ambiguous
rejectStage 1 failed, smoke failed, or model incompatible (no tool calling)

Optional Slack webhook notification fires on completion when configured.

12. Operating the daemon

Quick start

# Prerequisites: Node 20+, ES, GPU VM SSH, Buildkite token
cp config/smoke-full.json config/local.json   # edit SSH + VM
cp .env.example .env                          # add secrets
npm install && npm run build
./scripts/start-local.sh --daemonize \
  --clear-pending --enqueue-after-clear 'Qwen/Qwen2.5-7B-Instruct'

# Monitor
tail -f .smoke-logs/daemon.log
benchmarker-queue reasoning --config config/local.json --model <modelId>
CLI flagEffect
