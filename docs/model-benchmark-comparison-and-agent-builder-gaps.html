OSS Model Benchmark Comparison
& Agent Builder Support Gap Analysis

Four open-weight models, fully benchmarked end-to-end (Stage 1 vLLM performance → Stage 2 Kibana security eval suites → Stage 3 EIS reasoning) on 2×A100-80GB via the elastic-llm-benchmarker pipeline. This report compares them for treadmill-style coding/agentic work and dissects why every one of them would be blocked at discovery by the current Agent Builder baseline gate — then specifies the fixes.

The most important finding is not about the models — it's about the gate. All four models were benchmarked only because they were force-enqueued. Left to the automatic agentBuilderBaseline filter, the single best real-world model (Ornith, 3/3) would have been silently rejected by a filter that checks whether the string "instruct" appears in the model name. Two byte-for-byte-identical architectures score 3/3 vs 1/3 purely on post-training — a difference no static gate can see. The gate optimizes for the wrong signal.

5Treadmill coding fitness

Treadmill dispatches multi-step coding/agentic tasks: read context, plan, call one tool at a time, iterate against gates. The relevant axes are agentic reasoning (triage proxy), structured generation (esql proxy), tool-call reliability, and throughput (autonomous loops are throughput-bound, not latency-bound).

Recommended treadmill routing:
  • Planning / multi-step / verification tasks → Ornith-1.0-35B. Quality dominates; the 757 tps / 106ms-tail cost is acceptable for autonomous loops.
  • Fast single-shot codegen, boilerplate, format-fixing → Qwen2.5-7B-Instruct. 2.5× the throughput, passes structured-gen; just don't hand it multi-step reasoning.
  • Skip Qwen3.6-35B-A3B — it occupies a 35B VRAM slot to deliver sub-7B agentic quality. The MoE speed benefit (vs a dense 35B) doesn't offset the quality gap vs a dense 7B that's both faster and better.

8Deep gap analysis + fixes

Five gaps, ordered by impact. Each cites the exact code and the concrete fix. The unifying principle from §6: shift weight off static heuristics and onto cheap behavioral signals.

GAP 1Instruct detection is name-matching, not capability — and it rejects the best modelP0 · hard reject
Code

checkInstructVariant() — model-candidate-filter.ts:581–599 — hard-rejects any model whose id lacks instruct / chat / -it / _it.

Evidence

Ornith-1.0-35B (3/3, best model) and Qwen3.6-35B-A3B both fail this — yet Qwen3.6 literally ships a chat_template in tokenizer_config.json, and both serve correctly through vLLM's hermes parser. Naming convention ≠ capability, especially for OSS/base/reasoning models.

Fix
  • Detect the real signal: presence of a chat_template (in tokenizer_config.json or a standalone chat_template.jinja). If present → instruct-capable regardless of name.
  • Fall back to the existing inferToolCallingSupport() logic (already checks the vLLM parser) as a secondary capability signal.
  • Demote the name-match to a warning, never a hard reject. (Note: Ornith has no chat_template in tokenizer_config yet works — so the ultimate authority is a runtime chat-completions probe; see Gap 5.)
// enrich HfConfigJson with a chat-template flag, then:
private checkInstructVariant(model: ModelInfo): FilterRejection | null {
  if (!this.requireInstructVariant) return null;
  if (model.hasChatTemplate || model.supportsToolCalling) return null; // capability wins
  const named = ['instruct','chat','-it','_it'].some(s => model.id.toLowerCase().includes(s));
  return named ? null : { criterion:'instruct_variant',
    reason:'No chat_template, tool parser, or instruct/chat name marker', isHardRequirement:false }; // warn, not reject
}
GAP 2Parameter-count gate is blind to MoE active parametersP1 · wrong signal
Code

checkParameterCount() — model-candidate-filter.ts:556–579 — compares model.parameterCount (total) to the 7B floor. estimateParameterCountFromHfConfig() (agent-builder-baseline.ts:103–130) uses a dense formula and never reads num_experts / num_experts_per_tok even though the HfConfigJson.text_config type already declares those fields (lines 34–35).

Evidence

Qwen3.6-35B-A3B passes the 7B floor on 35B total but has ~3B active — and is beaten on RAG by the dense 7B. Total params over-state the agentic capability of MoE models by ~10×.

Fix
  • Compute active params when num_experts_per_tok is present: active ≈ dense_backbone + (expert_params × experts_per_tok / num_experts).
  • Apply the floor to a blended metric, or keep the total-param VRAM check but add an active-param warning when active < floor. Don't hard-reject on active alone — a well-post-trained 3B-active can still be excellent (Ornith).
GAP 3Context gate trusts advertised context; deployment silently runs nativeP1 · silent degrade
Code

checkContextSize() — model-candidate-filter.ts:459–469 — compares model.contextWindow to 128k. extractContextWindowFromHfConfig() already handles rope_scaling (lines 83–86), but enrichModelInfoFromHfConfig() takes Math.max(card, config) so a card advertising 131k passes even when native is 32k and no rope-scaling is configured.

Evidence

Qwen2.5-7B/14B are 32,768 native (config), advertised 131,072 (card, YaRN-only). The gate passes on the advertised figure; vLLM then deploys at 32k unless rope-scaling flags are explicitly set — the model qualifies at a context it isn't actually serving.

Fix
  • Track native and rope-scaled context separately. Gate on the context the deployment will actually serve.
  • If a model needs YaRN to hit 128k, have vllm-deployment.ts auto-emit the --rope-scaling / --max-model-len flags (so the passing gate matches reality), OR gate on native and warn.
  • Optionally lower the floor per-workload — a 32k model is fine for tasks whose context fits.
GAP 4100% tool-call success is unrealistic for small models — and the tiering pattern already existsP1 · over-strict
Code

minToolCallSuccessRate: 1.0 — src/types/config.ts:47 (and all config files). Meanwhile ITL is already tiered by parameter count via maxItlP50MsTiers (config.ts:64, resolveMaxItlP50Ms()) — the exact pattern that should apply here.

Evidence

Agent Builder issues single-tool calls, but smaller models occasionally emit malformed tool-call JSON. Demanding a perfect 100% success rate rejects otherwise-usable small models — especially since Agent Builder already compensates: validate_tool_calls.ts filters malformed args instead of 500-ing, and CHAT_MODEL_MAX_RETRIES retries transient failures.

Fix
  • Tier minToolCallSuccessRate by size, mirroring maxItlP50MsTiers (e.g. ≥30B → 0.98, 14–30B → 0.95, 7–14B → 0.90).
  • Score against post-repair success (after validate_tool_calls.ts filtering + one retry) so the metric matches production behavior, not raw first-attempt JSON validity.
GAP 5Qualification relies on static gates + security-domain evals — not a cheap behavioral coding probeP2 · strategic
Observation

§6 proves static gates can't rank models. Stage 2 is three security suites; for coding/agentic support they over-index on security-domain knowledge and under-weight tool-use trajectory. Every model passes esql; only Ornith passes triage — but neither tells you much about SWE-style coding.

Fix (fills the gap §6 exposes)
  • Add a fast behavioral pre-check before the expensive gate: deploy → single-tool chat-completions probe (5–10 tool-call prompts) → measure post-repair success + basic instruction-following. This is the only signal that separated Ornith from Qwen3.6, and it costs minutes, not the full CI matrix.
  • Add a coding/agentic eval suite alongside the security suites (SWE-bench-lite or a tool-use-trajectory suite) so coding models are qualified on coding tasks.
  • Agent Builder content side: smaller models exhibit the "split signal" (skill loaded, prescribed tools skipped). Add an imperative Tool-Call Contract to SKILL.md content and trim tool-schema verbosity so small-context models aren't overwhelmed by the system prompt.

Bottom line. The benchmarker works — it correctly identified Ornith-1.0-35B as a "support" model and correctly graded the others. The discovery gate in front of it is the weak link: as written it would reject all four models, including the best, on heuristics (name strings, total-param counts, advertised context) that §6 proves cannot predict agentic quality. Fixes #1 and #2 are small and unblock the most capable OSS models immediately; #5 is the durable answer — a cheap behavioral probe is the only thing that actually separates a great model from a mediocre one of the same size.
