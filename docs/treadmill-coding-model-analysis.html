Is Ornith-1.0-35B the best coding model now?

Deep analysis + recommended models for treadmill coding tasks · compiled 2026-07-06 · sources: DeepReinforce, HF, vals.ai/ModelFit, Morph, Kilo, and our own elastic-llm-benchmarker runs

Short answer: for its size class (35B) and specifically for agentic / terminal coding — which is exactly what treadmill does — yes, on current evidence. Ornith-1.0-35B leads every 35B-tier benchmark the vendor publishes, and — more importantly — it independently passed our agentic security-eval suite at 100% on all three suites with the best latency we've measured. It is one of only two models we've ever graded SUPPORT.

But "best" depends on the axis. For raw single-shot patch generation (SWE-bench Verified), the dense Qwen3.6-27B (77.2) narrowly edges Ornith-35B (75.6) and runs on smaller hardware. Ornith's real edge is terminal/agentic (Terminal-Bench 2.1: 64.2 vs Qwen3.6-35B's 52.5) — the loop treadmill runs. And every Ornith coding number is vendor self-reported; treat them as an upper bound, not a guarantee.

1. What Ornith actually is

deepreinforce-ai/Ornith-1.0-35BMoE · qwen3_5_moe~35B total262K ctxMITreleased Jun 2026post-trained on Qwen 3.5

Ornith-1.0 is a family of open agentic coding models (9B-Dense, 31B-Dense, 35B-MoE, 397B-MoE) from DeepReinforce, trained with a "self-scaffolding" RL approach where the model generates its own task harnesses. The 35B is the mid-tier MoE. It ships in bf16, FP8, and GGUF — so it runs locally via llama.cpp / LM Studio / Ollama, which matters for treadmill's local execution path.

Caveats (read before trusting the numbers):
  • Self-reported. All Ornith coding scores are vendor figures under DeepReinforce's own recipe (specific harness, temp=1.0, modified chat template). No independent lab has reproduced the vendor numbers yet.
  • Harness/quant sensitive. An independent run (NEO, Q4_K_M on a single RTX 3090) explicitly notes local numbers differ from vendor and "are not directly comparable" — quant + sampling + harness swing results materially.
  • Our 100%×3 is a small-N suite tuned for security tool-calling, not a broad coding benchmark — strong corroboration of agentic reliability, not proof of coding SOTA.

5. Recommended models for treadmill — by role

Treadmill routes work: task execution / fast → Hermes → LM Studio (local, localhost:1234); plan-gen / plan-chat / PR-review / verification → GitHub Copilot Claude (cloud). The list below maps to those roles and to your local hardware (M4 Max 48GB dev; M1 Max 64GB LAN appliance). MoE models with ~3B active params are the latency sweet spot for local execution.

Role A — Local agentic task execution (where the choice matters most)

#1 Ornith-1.0-35B new default

MoE ~3B act · MIT · 262K · GGUF/FP8
  • Best 35B-tier agentic/terminal scores; our benchmarker: SUPPORT/high, 100%×3.
  • 4-bit ≈ 18–20 GB → fits 48GB with headroom; MoE = fast decode.
  • Purpose-built for terminal/self-scaffolding agents = treadmill's exact loop.

#2 Qwen3-Coder-30B-A3B safe / proven

MoE 3B act · Apache 2.0 · 262K
  • Code-specialized, battle-tested; already the family treadmill routes to.
  • Lower risk than Ornith (Apache, widely reproduced) — good A/B baseline.
  • Keep as fallback when Ornith regresses on a task type.

#3 Qwen3.6-27B (dense) best raw patch

dense · Apache 2.0 · 262K
  • Top open-weight SWE-Verified (77.2); edges Ornith on pure patching.
  • Dense → slower decode than MoE, but fits 24GB.
  • Use for patch-heavy / single-shot fix tasks over long agentic loops.

alt Devstral 2 24B

dense · Apache 2.0
  • Agentic-coding specialist, 72.2 SWE-Verified, small footprint.
  • Good third option if you want a non-Qwen lineage for diversity.

Ornith-1.0-9B (dense)

  • Ideal for the "fast" router lane where 35B is overkill.

Role C — Plan-gen / PR-review / verification (keep cloud frontier)

Claude Opus 4.8 / GPT-5.x — no change

cloud · already treadmill's default for these roles
  • Cloud frontier is still ~10–12 pts ahead on SWE-Verified (Opus 4.8 = 87.6, Fable 5 = 95.0).
  • Judgment-heavy roles (planning, reviewing, gating) are where that gap pays off — don't localize these.
  • Local models are for high-volume execution, not final review.

6. Concrete recommendation

  1. Promote Ornith-1.0-35B to treadmill's local task-execution default (LM Studio, 4-bit MLX/GGUF), with Qwen3-Coder-30B-A3B as the fallback. Both are MoE-fast and fit your 48/64 GB boxes.
  2. A/B it for one week against Qwen3-Coder on real treadmill jobs — measure task pass-rate + retries, not just benchmark scores, since Ornith's coding numbers are vendor-only.
  3. Keep Copilot Claude for plan-gen / review / verification — the cloud gap is real and those roles are low-volume.
  4. Optional: benchmark Ornith-1.0-9B for the fast lane and Qwen3.6-27B for patch-heavy tasks, then let treadmill's router pick per task type.
