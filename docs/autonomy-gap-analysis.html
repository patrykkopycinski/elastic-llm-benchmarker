24/7 Autonomy Gap Analysis

What stands between elastic-llm-benchmarker and unattended, self-sustaining model benchmarking.

Generated 2026-07-04 · scope: discovery → queue → benchmark → CI-eval → Stage 3 → forward

Executive summary

The pipeline runs correctly end-to-end today, but it is operator-driven, not autonomous. A human enqueues models, watches the daemon, and restarts it after crashes. This session closed the two hardest correctness gaps (queue state-machine races and stale model selection). Three operational gaps remain before the daemon can run for weeks unattended: process supervision, discovery-on-by-default, and cost/lifecycle guardrails.

P0 blockers — required for unattended operation

1. Process supervision & auto-restart

P0Missing

scripts/start-local.sh ends in exec "${CLI}" start … — the daemon runs in the foreground of whatever shell launched it. If it crashes (OOM, unhandled rejection, ES 5xx storm), nothing restarts it. The VM keeps burning GPU-hours with no benchmark running. There is no launchd plist for the benchmarker (unlike patryks-treadmill, which has one).

Fix sketch
Ship com.elastic-llm-benchmarker.plist (launchd, KeepAlive=true, ThrottleInterval) analogous to treadmill's. Wrap the exec in a run-daemon.sh that tail-truncates the log on boot. Add treadmill install-style ./scripts/install-daemon.sh. Driver: scripts/install-daemon.sh (new). Acceptance: kill -9 $(cat .smoke-logs/daemon.pid) → daemon back within ThrottleInterval, lease re-acquired, active entry reclaimed to pending.

2. Discovery loop on by default

P0Missing

discoveryScheduler.enabled: false in config/local.json. The daemon only benchmarks what a human enqueues. "Keep benchmarking promising models" is impossible while discovery is off — the recency/supersession logic shipped this session never runs. Current settings (when enabled): intervalMinutes: 1440, maxModelsPerRun: 1, minTrendingScore: 30, autoQueue: true.

Fix sketch
Flip discoveryScheduler.enabled: true in config/local.json. Validate one cycle auto-queues a fresh, non-superseded, hardware-fit candidate that passes the Agent Builder baseline. Keep maxModelsPerRun: 1 and maxConcurrentRuns: 1 (one model on the VM — user invariant). Acceptance: node dist/cli.js discovery-run --once enqueues exactly one pending entry and logs skipped superseded siblings.

3. Cost / daily budget cap

P0Missing

There is no spend ceiling. A hung deploy or a runaway retry loop on an always-on VM accrues GPU cost with no automatic brake. The only bound today is a per-run max-duration in the worker — nothing caps cumulative daily spend or number of runs. Because the VM must never be stopped, the guardrail has to be queue-side (stop dequeuing), not VM-side.

Fix sketch
Add budget: { maxRunsPerDay, maxRunHours, pauseQueueWhenExceeded } to config. In the scheduler poll, before dequeue(), sum completed+active runs in the trailing 24h window from benchmarker-results; if over cap, skip dequeue and log a budget_paused event (do not touch the VM). Acceptance: with maxRunsPerDay: 2 and 2 completed today, the 3rd pending entry stays pending and a budget_paused row lands in benchmarker-errors.

P1 hardening

4. VM never-stop runtime guard

P1Partial

The invariant ("stopping the VM loses it permanently") is now an authored rule, but nothing in code prevents an accidental stop path. If any future teardown code calls a stop/terminate on the VM provider, the rule won't save it.

Fix sketch
Grep for any provider stop/terminate call; assert none exist on the happy path. Add a vmLifecycle: { allowStop: false } config gate that any teardown helper must check and refuse. Acceptance: a unit test asserts the SSH/VM service exposes no reachable stop call while allowStop: false.

5. Failure alerting

P1Missing

Errors land in the benchmarker-errors index but nothing pages a human. A stuck Buildkite poll, a repeated deploy failure, or a budget pause is invisible until someone opens the dashboard.

Fix sketch
Add a lightweight Slack webhook forwarder for severity: critical error docs (repeated deploy failure, lease-lost, budget-paused, discovery-empty-N-cycles). Reuse the existing tunnel/notify plumbing. Acceptance: forcing a deploy failure posts one Slack message with the model id + error.

6. Retry cap / dead-letter on repeat failure

P2Partial

reclaimStaleEntries() resets orphaned entries to pending — correct for a single crash, but an entry that fails deterministically (bad arch, VRAM overflow) will loop forever: dequeue → fail → reclaim → dequeue. There is no attempt counter.

Fix sketch
Add attempts to the queue entry; increment on each dequeue; when attempts > maxAttempts (e.g. 3), move to a terminal dead_letter status (distinct from transient failed) so reclaim/dequeue skip it. Acceptance: a model that fails deploy 3× lands dead_letter and is never re-dequeued.

Recommended rollout order

  1. P0-1 supervision — without auto-restart, everything else is moot the first time the daemon dies.
  2. P0-3 budget cap — before turning discovery on, cap spend so an always-on VM + auto-queue can't run away.
  3. P0-2 discovery on — flip the switch once supervision + budget are in place; this is what makes it "keep benchmarking".
  4. P1-5 alerting — so unattended failures reach a human within minutes.
  5. P2-6 retry cap + P1-4 VM guard — polish for long-run stability.
One-model invariant preserved throughout. Every guardrail is queue-side (maxConcurrentRuns: 1, dequeue gating). Nothing here stops or terminates the VM — the budget brake pauses dequeuing, not the machine.
